1. CAUSA RAÍZ IDENTIFICADA
================================================================================

NO es un bug de "sobrescritura de invitation_content al cambiar tema".

updateThemeSelection() está LIMPIO:
✅ Solo toca invitations.theme_id
✅ Preserva 100% de invitation_content
✅ No llama a updateProtagonists, updateHero, updateDressCode, etc.

LA CAUSA PROBABLE:
El problema reportado (datos que "desaparecen") es probablemente un bug de
RENDERIZADO, no de base de datos:

1. Datos personalizados SÍ se guardan en invitation_content
2. Pero componentes como Countdown, Hero, etc., pueden estar usando theme
   como fuente principal, ignorando datos personalizados en invitation_content
3. Cuando cambia tema, el fallback visual del tema cambia
4. Usuario percibe que se "perdió" el dato, pero en realidad está en BD

5. PROBLEMA PROBABLE - RENDERER
================================================================================

Si datos personalizados no se RENDERIZAN después de cambiar tema:

Contador (Countdown.tsx):
- background: 'var(--v2-countdown-card-bg-top, <fallback>)'
- Si Countdown no verifica invitation.countdown.customBackground
- Siempre muestra fallback del theme

Hero.tsx:
- Si não verifica invitation.hero.customBackground
- Muestra fondo del theme

Dress Code.tsx:
- Si não verifica invitation.dressCode.colors
- Muestra colores del theme

SOLUCIÓN: Implementar prioridad en cada componente:
1. Si existe invitation.SECTION.customField → usar ESO
2. Si no → usar theme.fallback
3. Si tampoco → usar fallback genérico
